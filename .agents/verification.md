# 验证规则

按改动类型选择最小但充分的验证，并如实报告未执行或无法执行的项目。

## 基础检查

所有变更完成后执行：

```sh
git diff --check
git status --short
```

只检查本次变更涉及的文件；发现用户已有的无关改动时，保留并在汇报中标注。

## 代码与发布链路

| 变更类型                            | 必要验证                                         |
| ----------------------------------- | ------------------------------------------------ |
| `src/` 实现、演示或类型改动         | `pnpm check` 与 `pnpm lint`                      |
| 任何公共 API、Rollup 或发布配置改动 | `pnpm build`，检查 `types/index.d.ts` 和产物名称 |
| `docs/` 改动                        | `pnpm docs:build`                                |
| README、规则或配置文本改动          | `pnpm format:check` 与基础检查                   |

构建会写入被追踪的 `types/index.d.ts`。若其变化不是本次公共 API 变更的预期结果，停止并报告，不要静默保留或丢弃。

`ci.yml` 必须与上述本地质量检查保持一致；`deploy-docs.yml` 只负责构建并发布 `docs/.vitepress/dist`。

## 汇报要求

汇报应区分：已执行并通过、已执行但失败、未执行及原因。当前没有自动化测试，不得将其他检查替代为测试结论。
