# 决策门槛

先判断变更是否影响消费者可观察到的行为、发布内容或协作历史；无法确定时按“需要确认”处理。

## 可直接执行

- 不改变公共 API、DOM 可见行为、浏览器支持范围或构建产物的局部重构。
- 明确的拼写、注释和与既有行为一致的 README 修正。
- 已确认方案中的规则文件维护，以及相关格式、类型、构建和 Git 状态检查。
- 在任务涉及的文件内补充验证，不删除用户已有的未提交改动。

## 必须先确认

### 产品与公开契约

- 改变水印文字、绘制、DOM 插入位置、样式层级、尺寸更新、观察策略、销毁语义或性能特征。
- 新增、删除或修改导出、参数、返回值、类型、打包入口、声明文件或兼容性承诺。
- 引入或修改数据持久化、埋点、用户数据、缓存数据、网络请求、请求封装、鉴权、重试或错误策略。

### 当前不存在的基础设施

项目当前没有权限、路由、菜单、缓存 key、组件 name、公共组件、请求封装或全局样式体系。首次引入它们，或以后修改它们的对外约定，必须先确认。

### 工具链与协作历史

- 升级、增加或删除依赖；修改 pnpm 版本、锁文件、Rollup、TypeScript、API Extractor、发布内容或浏览器兼容策略。
- 大规模删除、目录迁移、批量重命名、从 npm 迁移工具配置，或修改生成产物的归属。
- `git commit`、推送、打 tag、创建发布版本、发布 npm 包、改写 Git 历史。

### 存在多种合理方案

先列出可选方案、对 API/兼容性/维护成本的影响和推荐项，等待选择；不要以默认偏好代替确认。

## 新问题沉淀规则

发现新的典型问题时，不直接把它写入长期规则。先按以下格式报告并等待确认：

1. 场景
2. 风险
3. 建议规则
4. 建议写入的 `.agents/` 文件

只有获得确认后，才能把该规则写入对应文件。
